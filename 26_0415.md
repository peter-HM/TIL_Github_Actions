# Composite Action (복합 액션)

## Authoring ?

- 직접 Actions를 만드는 것이다. > 이제까지는 타인이 만든 actions를 marketplace에서 가져와서 사용함. > 왜? > 중복되는 step들이 여러 job에서 반복될 떄.

## Compostie Action ?

- 여러 step을 묶어서 하나의 Action으로 만드는 것 > 재사용이 용이하게 하기 위해
- uses: ./06-authoring-action/composite-action > 내 저장소에서 가져옴

- hello-world-1, hello-world-2 모두 같은 composite-action을 사용 with 값을 다르게 넘김
- hello-wolrd-1 > who-to-greet: "from composite action in hello-world-1!"
- hello-world-2 > who-to-greet: "from composite action in hello-wolrd-2!"

### 참조

- ./06-composite-action/action.yaml를 참조하여 사용하고 있음
- 즉, workflow.yaml에서 with: who-to-greet: "hello"를 action.yaml의 inputs: who-to-greet를 가져와서 사용하고 있고 outputs:random-number을 outputs.random-number로써 결과값으로 받고 있는거다.

#### action.yaml 맛보기

```

inputs:
    who-to-greet:
        description: "who to greet"
        required: True
        default: "world"

outputs:
    radom-number:
        value: ${{ steps.random-number-generator.outputs.random-number }}

runs:
    using: "composite"
    steps:
        - name: Set Greeting
          env:
            INPUT_WHO_TO_GREET: ${{ inputs.who-to-greet }}
          run: echo "Hello $INPUT_WHO_TO_GREET"

        - id: random-number-generator
          run: echo "random-number=$(echo $RANDOM)" >> $GITHUB_OUTPUT

```

- random-number-generator Step의 output > outputs.random-number로 등록 > workflow.yaml에서 steps.composite-action-instance.outputs.random-number로 받음

# Reusable Workflow

- composite action과의 차이점?
- composite action > 여러 stpe을 묶어서 재사용함
- reusable workflow > 워크플로우 전체를 재사용


Composite Action

- step들을 묶어서 재사용함
- uses: ./path
- job은 포함하지 않으며 stpe만 포함


Reusable Workflow

- workflow 전체를 재사용함
- uses: ./path + workflow_call
- job을 포함한 workflow 전체
- 공통 배포 파이프라인을 묶을 떄 사용한다.

## workflow_call ?

```

on:
    workflow_call:

```

- 사람이 수동으로 실행하는 workflow_dispatch와 달리 workflow_call은 다른 workflow가 호출해서 실행하게 된다.

'''

on:
    workflow_call:
        inputs:
            foo:
                required: true
                type: string
            environment:
                required: true
                type: string
        secrets:
            EXAMPLE_REPOSITORY_SECRET:
                required: true

'''

- composite action의 inputs: 와 같은 개념 > 호출하기 위해서는 해당 값들을 넘겨줘야해~

- 또한 해당 예제의 주석이 중요하다. > #NOTE: you can only pass repo/org secrets here, not environment secrets > secret에는 Repository/Organlization 레벨의 secret만 넘길 수 있다. 즉, Envrionment Secret은 넘길 수 없고 아래 처럼 environment를 지정하면 자동으로 가져온다.

```

jobs:
    reusable-workflow-job:
        environment: ${{ inputs.environment }} # 여기서 environment 지정
        steps:
            - env:
                EXAMPLE_ENVIRONMENT_SECRET: ${{ secrets.EAMPLE_ENVIRONMENT_SECRET}}
                # environment 지정하면 자동으로 접근이 가능

```

#### secret:inherit vs 직접 지정

- secrets: inherit > 나의 secret 전부 넘겨줄게 > 편리하미나 불필요한 secret도 넘어갈 수 있다.
- secrets: EXAMPLE_REPOSITORY_SECRET: ${{ secret.XXX }} > 불필요한 것만 골라서 넘길게? > 보안상 훨씬 안전

## Submodules ?


Git 저장소 안에 다른 Git 저장소를 포함시킬 수 있다. submoules: true 는 그 안에 포함된 저장소도 함꼐 가져오라는 의미이다.

내 저장소
└── code
└── 외부 저장소 A (submodule)
└── 외부 저장소 B (submodule)

## 3 jobs 흐름

3가지 job 모두 공통된 흐름을 가지고 있음

```

# 1. 코드 가져오기
- uses: actions/checkout@v4...

# 2. Action 실행
- uses: ./path/javascript-action
  id: hello
  with:
    who-to-greet: "..."

# 3. output으로 받아서 실행
- run: echo "The time was ${{ steps.hello.outputs.time }}

```

- time을 output으로 내보내는 건 Javascript 코드 안에서 처림함. > Composite Action과 동일하게 GITHUB_OUTPUT에 저장한다.

# Container Action

Action 종류 비교

- Composite Action > YAML로 Step 묶기
- Javascript Action > Javascript 코드로 만들기
- Container Action > Docker container로 만들기
- Reusable Workflow > workflow 전체 재사용

## Container Action ?

Docker container안에서 Action을 실행하는 것이다.

- javascript action > Node.js 환경에서만 실행가능
- Container action > 어떤 언더든 실행 가능하다. > Docker container가 실행 환경 자체를 통쨰로 만들어주기 떄문에 언어 제한이 없다.

## Job 3가지 비교

### Job 1 - shell + Dockerfile

```

uses: ./shell-container-actions

```

- 실행할 떄, Dockerfile을 읽어서 container를 직접 build > 느리지만 항상 최신 상태

### Job 2 - shell + Image 사용

```

uses: ./shell-container-action-prebuild

```

- 이미 만들어진 container images를 가져다가 사용 > 빌드 과정이 없기에 빠르다.

### job 3 - python + Dockerfile

```

uses: ./python-container-action

```

- Python으로 작성된 Action을 container로 실행 > javascript Action으론 Python 코드 실행 불가 > Container Action으로 가능


## Dockerfile Build VS Image 사용

Dokerfile Build
- 느린 속도
- 간단한 관리
- 개발할 떄 주로 사용


Image 사용
- 빠른 속도
- 이미지 따로 관리가 요구됨
- 배포할 떄 사용
