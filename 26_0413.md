run : |

uses: action/upload-artifact@...

##### CONSUME #####
uses: action/download-artifact@...
with:
    name: example-artifact

external service auth
2 primary mechanisms to authernticate to external systems

1) Static credentials (e.g. API Key)

with:
    aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }} 

- long-lived credential stored in GHA repo/environment secret
- Less secure
- Compatible with basically everything
- AWS에서 발급한 고정된 pw를 secret에 저장해두고 사용하는 방식
- cons:
    비밀번호 유출되면, 큰일
    pw 만료 관리, 사람이 직접 주기적으로 교체
    pw 여러 곳에 쓰면? 관리가 엉망

2) OIDC(OPEN ID CONNECT) Token

permission:
    id-token: write     #JWT 토큰 발급 권한
with:
    aws-region: us-east-2
    role-to-assume: arn:aws:iam::9155123124:role/github-action-role

- Short-lived credential retrieve at runtime
- More secure
- Requires service to suppoert OIDC
- Github, AWS가 비밀번호 없이 서로 신뢰하는 방식이다.
Github Action 실행 > 
Github: 이 workflow한테 임시 토큰 줄게 (JWT) >
AWS: Github가 보증하네? 임시 접근 통과~ >
workflow가 AWS 사용 (끝나면 토큰 자동 만료)


In AWS you can use both of them to manage permission. 
But, OIDC is the method they recommended
