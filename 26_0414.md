<h2>Matrix and conditions</h2>


<h3>Matrix?</h3>

같은 job을 여러 조합으로 동시에 실행하는것
matrix:
    number: [1, 2]
    letters: [a, b, c]

이렇게 되면 자동으로 모든 조합을 만들어 준다.
1a, 2a, 1b, 2b, 3a, 3b  > 이 조합들이 동시에 병렬로 실행됨

<h3>Exclude?</h3>

특정 조합을 제외하고 실행하게 한다.
exclude:
    - number: 1
      letter: c

1a, 1b, --1c--, 2a, 2b, 2c

<h3>Conditions</h3>

if: ${{ ! (matrix.number == 2 && matrix.letter == 'c') }}

2c 일때만 이 step을 건너뛴다. > 1a, 1b, 2a, 2b, 2c(가 되는 순간 건너뜀)

<h3>Fail-fast</h3>

strategy:
    fail-fast: true

조합 중 하나라도 실패하면 > 나머지 실행 중인 조합 전부 즉시 중단 > false 값인 경우, 하나 실패해도 나머지는 계속 실행

<h3>cConcurrency</h3>

concurrency:
    group: ${{ github.workflow }}-${{ github.ref }}
    cancel-in-progress: true

같은 workflow가 이미 실행 중이면 취소하고 새로 실행하라
workflow 실행 중 > 같은 workflow 또 실행됨 > 기존 것 취소 > 새로운 것을 실행

<h3>sleep-job</h3>
- run: sleep 100
  timeout-minute: 2

예제에 존재하는 이유 > 의도적으로 오래 실행시켜서 취소되는 걸 보여주기 위해 > concurrency test를 위해.


ex) Matrix를 쓰는 경우
node.js 16, 18, 20 version에서 모두 테스트
ubuntu, window, macOS에서 모두 테스트

matrix:
    os: [ubuntu, windows, macos]
    node: [16, 18, 20]

<h2>Dynamic Matrix</h2>

<h3>기존 matrix와의 차이</h3>
기존 matrix > 조합을 코드에 직접 고정해서 작성

- 코드에 조합을 고정
- 유연성이 낮음
- 조합이 정해져 있을떄, 주로 사용

Dynamic matrix > 조합을 실행할 떄 동적으로 생성

- 실행할떄, 조합이 생성
- 유연성이 높음
- 조합이 매번 달라질 떄, 주로 사용함.

- 즉, 조합이 실행 시점에 결정이 되어 코드 작성 시점에는 조합을 알 수 없는 경우에 사용
- API 서버 목록이 매번 달라지는 경우, 배포할 환경이 동적으로 결정되어 질떄, 사용자가 직접 테스트 범위를 정할떄!!


전체 흐름 
- 사용자가 숫자 입력 (ex. 3) > generate-matrix job: 3쌍의 랜덤 값을 생성 > execute job : 생성된 조합으로 Matrix 실행

<h3>Example</h3>

<h4>Inputs</h4>

 ```
on:
    workflow_dispatch:
        inputs:
            length:
                description: "Number of foo/bar pairs to generate
                required: True
                default: "2"
 ```

- Github action > workflow_dispatch > "몇 쌍 생성할까요?" 입력창이 뜨게됨 > 사용자가 숫자 입력 > ${{ github.event.inputs.length }} 로 받아옴

<h4>generate-matrixJob</h4>

```

env:
    LENGTH: ${{ github.event.inputs.length }}

n= "${LENGTH:-2}"

for ((i=1; i<=n; i++)); do
    foo_val= $((RANDOM % 100))
    bar_val= $((RNADOM % 100))
done

>> result

[
    {"foo": 42, "bar": 17},
    {"foo": 83, "bar": 55},
    {"foo": 11, "bar": 73}
]

```
해당 resultrk GITHUB_OUTPUT에 저장되어서 다음 job에 전달된다.

<h4>executeJob</h4>

matrix:
    include: ${{ fromJSON(needs.generate-matrix.outputs.include) }}

fromJSON : 문자열로 전달된 JSON을 실제 데이터로 변화해준다.

다음과 같이 실행된다.
조합 1: foo=42, bar=17
조합 2: foo=83, bar= 55
조합 3: foo=11, bar= 73

<h2>Workflow Commands</h2>

- Github Actions가 제공하는 특별한 명령어들
- echo "::cmd::content" 형태로 신호를 보내는 것

<h3>Log - level (annotations)</h3>

echo "::debug::Step started" > 디버그 (숨겨져 있고, 설정해야 보임)

ehco "::notice::msg" > 파란색 알림

echo "::warning::Warning msg" > 노란색 알림

echo "::error::error msg" > 빨간색 알림, 시각적으로만 표기될 뿐, 실행을 멈추지는 않는다, 이를 위해선 exit 1를 사용해야 함

<h3>Log-group</h3>

echo "::group::log_group_name"

echo "content in group"

echo "::endgroup"

- 로그가 많을 때, 접었다 펼 수 있게 그룹으로 묶는다.

<h3>값 masking</h3>

echo "::add-mask::$FOO"

echo "FOO is $FOO" >> "FOO is ****"

- Secret 처럼 로그에서 값을 숨길 수 있다. 동적으로 생선된 민갑한 값을 숨길 떄 사용한다.

<h3>workflow cmd stop/resume</h3>

marker= $(uuidgen) > 랜덤 고유값 생성

echo "::stop-commands::$marker" > 여기서부터 명령어를 무시 시작

echo "::error::이건 에러로 취급하지 않음"

echo "::$marker::" > 여기까지, 그리고 여기서부터 cmd 다시 인식

echo "::notice::다시 동작" > 정상동작

- 특정 구간의 cmd를 의도저긍로 무시하고 싶을 떄 사용한다.

<h3>Redirection</h3>

echo "BUILD_ID=${date}" >> "$GITHUB_ENV"

echo "choice=green" >> "#GITHUB_OUTPUT"

- 환경변수 설정 및 step output에 저장

<h3>Step Summary</h3>

echo "#### Build completed" >> $GITHUB_STEP_SUMMARY

echo "- Build ID: $BUILD_ID" >> $GITHUB_STEP_SUMMARY

echo "- Colour: ${{ steps.pick-color.outputs.choice }}" >> $GITHUB_STEP_SUMMARY

<h3>PATH 추가</h3>

echo "$HOME/.local/bin" >> "$GITHUB_PATH"

- 커스텈 프로그램 경로를 PATH에 추가해서 다음 Step부터 그 프로그램을 바로 실행할 수 있게 한다.
