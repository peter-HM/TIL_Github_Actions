# Honey tips

```

# 방식 1 - 여러 줄 명령어
- run: |
    echo "첫번째"
    echo "두번째"
    echo "세번째"

# 방식 2 - 각각 별도 Step
steps:
  - run: echo "첫번째"
  - run: echo "두번째"
  - run: echo "세번째"

```

방식 1과 2의 차이

방식 1
- 하나의 step으로 실행되는 것
- 하나가 실패할 경우 전체가 중단됨
- 로그 또한 한 덩어리로 나오게 됨
- 관련된 명령어를 묶어서 사용할떄

방식 2
- 각각 별도의 step
- 하나가 실패하더라도 다음 step 실행이 가능하다.
- 로그는 각 step별로 구분되어 나옴
- 독립적인 명령어를 구분하여 사용할떄

# Workflow Iteration - SSH Breakpoint

- Main idae > workflow가 실패했을떄, 그 환경에 직접 ssh로 접속해서 debugging하는 방법

## STEP

### Step1 - 코드 가져오기

```
- uses: action/checkout@...

```

### Step2 - 의도적으로 실패

```

- name: Fail
  run: |
    echo "::error Purposefully failing the job" # Print error log "
    exit 1   # 강제로 실패시키기

```

- ::error:: > workflow Commands로 에러 표시
- exit 1 > bash에서 "실패로 종료해라"

### Step3 - SSH BreakPoint

```

- name: Breaking if failed
  if: failure()    # 이전 step이 실패했을떄,
  uses: namespacelabs/breakpoing-action@...
  with:
    duration: 10m   # 10분간 ssh 접속 유지
    authorized-users: sidpals  # 이 github user만 접속가능

```

#### SSH BreakingPoint ?

일반적인 실패
- 로그 보고 원인 파악
- 재현하기 어려운 경우 답답함

Ssh BreakingPoint
- 실패한 그 순간의 container에 접속
- 파일 확인, 명령어 직접 실행
- 원인 파악에 용이

사용 예시
- 로컬에서는 잘 되는데, CI에서만 실패할 떄
- 환경 차이로 인한 버그를 debugging 할 때
- 재현이 어려운 문제를 직접 확인할 때


# Workflow Iteration - debug 환경 함수

- Main Idea > 로그 레벨을 조절해서 필요할 때만 상세 로그를 볼 수 있다.

## Log Level

```

echo "::debug::This is a debug message"    # 기본적으로 숨겨짐
echo "::info::This is an info message"     # 항상 보임
echo "::warning::This is an warning message" # 항상 보임 (노란색)

```

## Debug log 3 step

1단계 > 기본 실행
- debug 로그 안 보임
- info, warning만 보임

2단계 > ACTIONS_STEP_DEBUG= true
- 내가 작성한 ::debug:: 로그가 보임

3단계 > ACTIONS_RUNNER_DEBUG= true
- runner 자체의 상세 로그까지 전부 보인다.


## 설정 방법

github 저장소/settings/Secrets and Variable/Variables/ACTIONS_STEP_DEBUT= true 추가


왜 기본설정에서는 숨기는가?
- Debug 로그가 항상 보이면 > 로그가 너무 많아서 오히려 가독성이 떨어짐 > 필요할 때만 켜는 것이 효율적이다.



# CI/CD 공통 패턴 정리

## 1. 코드 검증 - Linting

Lint ?

- 코드가 규칙에 맞게 작성되어있는가 자동으로 검사하는 것
- 들여쓰기, 세미콜론 여부, 변수 확인, 버그를 유도할 코드가 있는가 등을 마치 맞춤법 검사기 처럼 하는 것.
- 실행하기 전 코드 품질을 검사한다.


1) Naive 방식  (Navie > 단순한, 최적화가 되지 않은)

```

jobs:
    lint:
        runs-on: ubuntu-24.04
        stpes:
            - uses: action/checkout@v4
              run: npm install eslint   # node.js의 린팅도구
              run: eslint .      # 전체 코드 린팅 (slow)

# python > pip install flake8
# Go > golangci-lint

```

2) Optimized 방식

```

jobs:
    lint:
        runs-on: ubuntu24-04
        steps:
            - uses: action/checkout@v4

            # 변경된 파일만 린팅
            - name: Get changed files
              id: changed-files
              uses: tj-action/changed-files@v44
              with:
                files: "**/*.js"

            # Super-Linter로 여러 언어를 한번에
            - name: Super Linter
              uses: super-linter/super-ilnter@v7
              env:
                GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
                VALIDATE_ONLY_CHANGED_FILES: true

            # PR에 결과 댓글로 달기
            - name: Comment PR
              if: failure()
              uses: action/github-script@v7
              with:
                script: |
                    github.rest.issues.crateComment({
                        issue_number: context.issue.number,
                        body: "Linting Failed. Please Check the code"
                    })

# script: > Javascript 코드로 Github API를 바로 쓸 수 있게 해주는 Action.

```

## 3. Build & Deploy

Push 방식 (직접 배포)

```

jobs:
    build-and-deploy:
        runs-on: ubuntu-24.04
        permission:
            id-token: write
        steps:
            - uses: action/checkout@v4

            # AWS authoring
            - uses: aws-actions/configure-aws-credentials@v4
              with:
                role-to-assume: arn:aws:iam::12234567:role/github-actions-role
                aws-resion: ap-northeast-2
            
            # Docker images build $ ECR PUSH
            - name: Build and push
              run: |
                docker build -t my-app
                docker tag my-app:latest $ECR_URL/my-app:${{ github.sha }}

            # ECS deploy (push)
            - name: Deploy to ECS
              run: |
                aws ecs update0-service \
                   --cluster my-cluster \
                   --service my-service \
                   --force-new-deployment

# github.sha > commit을 식별하는 고유 hash값
# git log > commit a1b2c3d4f5... 이것이 SHA
# docker 이미지 tag로 my-app:a1b2c3d > 이 커밋으로 빌드된 이미지 > 어떤 cmomit으로 빌드된 이미지인지 정확히 추적가능

```

GitOps 방식 (설정 파일만 수정)

```

jobs:
 upsdate-manifest:
    runs-on: ubuntu-24.04
    steps:
        - uses: actions/checkout@v4
          with:
            repository: my-org/k8s-manifest  # 설정 파일 저장소

        # 이미지 버전만 바꾸기
        - name: Update Image Tag
          run: |
            sed -i "s|image: my-app:.*|image: my-app:${{ github.sha }}" deploy.yaml

        # 변경사항 commit (ArgoCD 등이 감지해서 자동 배포)
        - name: Commit changes
          run: |
            git commit -am "Updata image to ${{ github.sha }}"
            git push

```

Push 방식은 workflow가 직접 배포, GitOps는 설정 파일만 바꾸면 배포 에이전트가 알아서 배포한다.


## 4. 저장소 자동화

### Release Please (릴리즈 자동화)

```

jobs:
    release:
        runs-on: ubuntu-24.04
        steps:
            - uses: google-github-actions/release-please-action@v4
              with:
                release-type: node

# 커밋 메시지 규칙
# feat: 새 기능     → Minor 버전 올림 (1.0.0 → 1.1.0)
# fix: 버그 수정    → Patch 버전 올림 (1.0.0 → 1.0.1)
# feat!: 破괴적 변경 → Major 버전 올림 (1.0.0 → 2.0.0)

```

### Stale 관리

```

jobs:
    stale:
        runs-on: ubuntu-24.04
        steps:
            - uses: action/stale@v9
              with:
                days-before-stale: 30  # 30일간 활동이 없으면
                days-before-close: 7   # 7일 후 자동 닫기
                stale-issue-label: "stale"
                stale-pr-lable: "stale"
     

```


#### stale ?

오래되어 더 이상 활성화되지 않은 이라는 의미.

- PR, Issue가 올라왔는데, > 30일동안 아무도 댓글도 안달고, 아무런 활동이 없으면 > "stale" 라벨 자동으로 붙임 > 7일 더 지나면 자동으로 닫기


언제 사용?

- 저장소에 PR, issue가 수백개 쌓이면, > 어떤 것이 유효한지 파악이 힘들다. > stale 자동화로 저장소를 깔끔하게 유지
